**EndpointStore.swift** - `@Observable final class EndpointStore`:
- Import SwiftUI, SwiftData, Foundation
- Private `let modelContext: ModelContext`
- `init(modelContext: ModelContext)` - stores the context
- Computed `var endpoints: [MockEndpoint]` - fetches all endpoints sorted by `\.sortOrder` ascending. Uses `FetchDescriptor<MockEndpoint>(sortBy: [SortDescriptor(\.sortOrder)])`. Returns `(try? modelContext.fetch(descriptor)) ?? []`.
- Computed `var endpointCount: Int` - uses `modelContext.fetchCount` with plain FetchDescriptor. Returns `(try? modelContext.fetchCount(descriptor)) ?? 0`.
- `func addEndpoint(_ endpoint: MockEndpoint)` - inserts into context, calls `try? modelContext.save()`
- `func deleteEndpoint(_ endpoint: MockEndpoint)` - deletes from context, calls `try? modelContext.save()`
- `func updateEndpoint()` - just calls `try? modelContext.save()` (properties are mutated directly on the model object, then caller calls this to persist)
- `func addLogEntry(_ log: RequestLog)` - inserts the RequestLog into context, saves, then calls `pruneOldEntries()`
- `private func pruneOldEntries()` - Fetches RequestLog count. If count > 1000, fetches ALL RequestLog sorted by `\.timestamp` ascending, then deletes entries from index 0 up to (count - 1000). Saves after deletion.

Note: EndpointStore handles both endpoint CRUD AND request log insertion/pruning. This keeps ModelContext usage centralized. Phase 2 will call `addLogEntry` when requests come in. If this feels too coupled, a separate RequestLogStore could be extracted later, but for now one store per ModelContext is simpler.

**ServerStore.swift** - `@Observable final class ServerStore`:
- Import Foundation
- `var isRunning: Bool = false` - tracks whether the mock server is currently running (set by Phase 2 server engine)
- `var port: UInt16` - get/set that delegates to `ServerConfiguration.port`. Initialize in init from ServerConfiguration.port.
- `var corsEnabled: Bool` - delegates to `ServerConfiguration.corsEnabled`. Initialize from it.
- `var autoStart: Bool` - delegates to `ServerConfiguration.autoStart`. Initialize from it.
- When setting port, corsEnabled, or autoStart, also write through to ServerConfiguration (which writes to UserDefaults).
- `var serverURL: String` - computed property returning `"http://localhost:\(port)"` (useful for display in Phase 3+)
- `init()` - reads current values from ServerConfiguration into local @Observable properties

Design: ServerStore holds local @Observable copies of the config values. When a property is set, it writes through to ServerConfiguration (UserDefaults). This way views get fine-grained @Observable tracking, and persistence is immediate. The `isRunning` state is purely in-memory (Phase 2 sets it).

**ProManager.swift** - `@Observable final class ProManager`:
- Import Foundation
- `static let shared = ProManager()` - singleton since PRO state is global
- `private(set) var isPro: Bool` - cached in-memory, backed by Keychain
- `private init()` - loads isPro from `KeychainService.loadBool(forKey: "isPro") ?? false`
- `func setPro(_ value: Bool)` - sets isPro, calls `KeychainService.saveBool(value, forKey: "isPro")`
- `func canAddEndpoint(currentCount: Int) -> Bool` - returns `isPro || currentCount < 3`
- `func canImportEndpoints(currentCount: Int, importCount: Int) -> Bool` - returns `isPro || (currentCount + importCount) <= 3`
- `static let freeEndpointLimit = 3` - constant for UI display

The singleton pattern is acceptable here because PRO state is app-global and must be consistent everywhere. It is injected into SwiftUI via `.environment()` for testability in views.
